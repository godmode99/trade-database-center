# รายงานประเมินโครงการ (ฉบับตั้งต้น)

อัปเดตล่าสุด: 2026-03-06

## 1) สรุปสถานะโดยรวม
- สถานะโปรเจค: **เริ่มต้นวางโครงงาน**
- เป้าหมายหลักชัดเจนแล้ว: ดึงข้อมูลเศรษฐกิจ/ตลาด, แปลงข้อมูล, จัดเก็บในฐานข้อมูล และแสดงผลผ่านเว็บ
- เทคโนโลยีหลักถูกกำหนดแล้ว (Next.js + Neon + Vercel + Prisma)
- ยังไม่มีรายละเอียดเชิงปฏิบัติของสคีมาฐานข้อมูล, API ภายใน, ตารางงานอัตโนมัติ, และเกณฑ์คุณภาพข้อมูล

## 2) ความครอบคลุมความต้องการข้อมูล
จากเอกสารโครงการ ความต้องการ data source ครอบคลุม 4 กลุ่มสำคัญ:
- **FRED**: daily/weekly/monthly พร้อมรายการ series ที่ชัดเจน
- **CME FedWatch**: probabilities (sofr/zq) + quotes หลาย timeframe
- **Calendar**: weekly snapshot + update เมื่อ actual ออก
- **MT5**: OHLCV หลาย timeframe + feature engineering เชิงเทคนิค

**ประเมิน:** ขอบเขตข้อมูลครบสำหรับงาน macro + rates + market structure ในเชิงใช้งานจริง

## 3) การอ้างอิงจากโปรเจคเก่า (ref)
ไฟล์อ้างอิงที่อ่านและเกี่ยวข้อง:
- `ref จากโปรเจคเก่า/python/fetch/fred/pipeline.py`
- `ref จากโปรเจคเก่า/python/fetch/mt5/pipeline.py`
- `ref จากโปรเจคเก่า/python/fetch/calendar/pipeline.py`
- `ref จากโปรเจคเก่า/python/fetch/cme_fedwatch/quotes_main.py`
- `ref จากโปรเจคเก่า/python/fetch/cme_fedwatch/probabilities/sofr_probabilities_main.py`

**ข้อสังเกตจาก ref:**
- โครง fetch pipeline ในโปรเจคเก่ามีแนวทาง retry, manifest, และ log ที่นำมาปรับใช้ต่อได้ทันที
- งาน CME มีความเสี่ยงเชิง browser automation/login จึงควรแยกส่วน monitoring และ fallback
- Calendar มีโจทย์ incremental update ที่ต้องระวังเรื่อง idempotency

## 4) ช่องว่าง/ความเสี่ยงสำคัญ
1. **สคีมาฐานข้อมูลยังไม่ถูกนิยาม** (ตาราง, ดัชนี, unique key, retention)
2. **ยังไม่ระบุ data contract** ระหว่าง fetch/transform/load และหน้าเว็บ
3. **ความน่าเชื่อถือของแหล่ง scrape** (โดยเฉพาะ CME/Calendar) อาจเปลี่ยน DOM หรือโดน rate limit
4. **ไม่มี SLO/SLA งานข้อมูล** เช่น freshness, completeness, latency
5. **ยังไม่เห็น test strategy** สำหรับ parser/transform และ validation ก่อนเขียนลง DB

## 5) ข้อเสนอแนะลำดับถัดไป
- กำหนด schema กลาง (raw + normalized + latest views)
- สร้าง job spec รายแหล่งข้อมูล (schedule, retry, timeout, alert)
- วาง data quality checks ขั้นพื้นฐาน (null ratio, duplicate, outlier)
- กำหนด API/read model ให้หน้าเว็บเรียกใช้โดยไม่ผูกกับ raw format
- เริ่มจาก vertical slice แรก: FRED daily → DB → หน้าเว็บ

## 6) สรุปผลการประเมิน
โปรเจคอยู่ในสถานะ **พร้อมเริ่มพัฒนาเชิงระบบ** แต่ต้องเร่งนิยามโครงสร้างข้อมูลและมาตรฐานคุณภาพก่อนเดินงานเต็มรูปแบบ เพื่อให้ขยายจาก ref เก่าได้อย่างปลอดภัยและดูแลต่อเนื่องได้ง่าย

## 9) อัปเดตการประเมินรอบล่าสุด (Database Design from Legacy Raw Data)
- มีการวิเคราะห์ payload จริงจาก legacy raw data แล้ว (ไม่ใช่แค่ requirement เชิงข้อความ)
- จัดทำเอกสาร `god_project/docs/DATABASE_DESIGN.md` ครอบคลุม logical model, table-level fields, unique keys, indexes และ ingestion traceability

**ผลต่อความพร้อมโครงการ:**
1. ลดความเสี่ยง schema drift ระหว่างทีม fetch กับทีม web เพราะมี field mapping ชัดเจนจากข้อมูลจริง
2. พร้อมเริ่ม Prisma schema v1 ได้ทันที โดยเริ่มจาก ingestion_runs + FRED + Calendar + MT5
3. มีแนวทางรองรับ incremental update ของ Calendar (actual ออกทีหลัง) และ idempotent upsert ของ time-series

**ช่องว่างที่ยังเหลือ:**
- CME FedWatch ยังต้องยืนยัน field จาก sample จริงก่อน finalize schema
- ยังไม่ได้แตกเป็น migration plan รายตาราง (ลำดับ deploy/change management)
